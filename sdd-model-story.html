Historia

7 semanas. Del prompt al proceso.

Cómo evolucionó el modelo desde el commit inicial (27 de mayo) hasta hoy.

Semana 7 · 5–14 julio
Métricas reales, seguridad y sdd-task
sdd-metrics pasó de estimar tokens (bytes÷4) a leerlos de verdad con ccusage en Claude Code. Después llegó el hardening: 6 remediaciones de una auditoría de seguridad (XSS en el kanban, bind a 127.0.0.1, checks de inyección en todo input no confiable). Y el cierre: sdd-task para implementar una task puntual con gate de dependencias, y el kanban documentado.
ccusage security hardening sdd-task kanban escapeHtml
Semana 5–6 · 22–29 junio
Visualización + TDD explícito + setup por IDE
El grafo de dominio ganó una visualización interactiva con vis.js — nodos por dominio, sidebar con detalle, distinción brownfield/greenfield. sdd-implement formalizó el loop TDD (Red → Green → Refactor) con gate de calidad estructural. sdd-setup se volvió IDE-aware con validación de tokens, y se documentó la regla de fallback MCP→REST con trazabilidad obligatoria.
vis.js graph TDD Red→Green→Refactor sdd-setup IDE-aware fallback MCP→REST
Semana 4 · 19 junio
El día grande: calidad de artefactos + Jira
Un solo día, dos frentes. Calidad: sección fuera-de-scope obligatoria en spec.md, tasks con slicing vertical (tracer bullets), prohibición de nombres de funciones, detección de tipo de feature en el refine, y el steering skill coding-standards para el equipo. Integración: los tres comandos Jira, sdd-setup, y sdd-generate creando tickets con jira-map.yaml.
fuera de scope v1 tracer bullets sdd-jira-* coding-standards skill jira-map.yaml
Semana 3 · 13–16 junio
Navegación de contexto + MCP server
Dos comandos para operar el contexto del agente: sdd-compact-guide (cuándo conviene compactar según la fase) y sdd-context-budget (cuánto pesa el framework en tokens). sdd-handoff sumó la sección obligatoria de caminos descartados y sdd-log el registro de alternativas. Además nació el sdd-mcp server con 11 tools determinísticas, que luego se movió a repo propio.
sdd-compact-guide sdd-context-budget sdd-mcp server caminos descartados
Semana 2–3 · 8–12 junio
Gobernanza de equipo + auditor determinista
El salto de un dev a un equipo. Registro maestro de features (features.yaml) con status OPEN/CLOSED, grafo de dominio para routing de contexto, detección de colisiones entre features de distintos owners, ruta corta sdd-fix para bugs. Y la pieza clave: sdd-audit.mjs, el auditor determinista que corre en CI sin IA — lo que el script verifica, los agentes no lo recalculan.
features.yaml sdd-audit.mjs + CI colisiones OPEN/CLOSED sdd-fix graph/domain.yaml
Semana 1–2 · 28 may – 8 junio
Telemetría DX + handoff + brownfield
Tres líneas en paralelo que convergieron en un merge el 8 de junio: la telemetría DX de patohed (sdd-metrics, rework ratio, DX_MET_001–006), el sdd-handoff de pmillanmc (snapshot de sesión para continuar en otro agente), y el soporte brownfield — sdd-scan lee el codebase existente y genera existing-arch.md, cuyas restricciones son no negociables.
sdd-metrics sdd-handoff sdd-scan rework ratio existing-arch.md merge 2 autores
Día 0 · 27 de mayo
Initial commit: el ciclo base
La idea fundacional: reemplazar prompts aislados por un proceso con fases, artefactos versionables y gates humanos. El ciclo original: refine → generate → implement → review. Cuatro comandos, cuatro artefactos (constitution, spec, plan, tasks), y el principio que sigue vigente: el código es la consecuencia del proceso, no el punto de partida.

Convención de branches (owner/tipo/feature_id), commits con feature_id como scope, template de PR y comando /sdd-pr-review que pre-revisa el diff contra la spec antes de que llegue al TL. El agente comenta pero nunca aprueba — el approve es exclusivo del TL.
